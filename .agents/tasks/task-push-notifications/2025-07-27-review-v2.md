# Firebase Cloud Messaging integration for IMS push notifications (v2)

This change adds push notification support to the IMS HR app via Firebase Cloud Messaging. The client registers an FCM service worker, obtains a device token on login, stores it in a per-user subcollection (`hrd_fcm_tokens/{userId}/devices/{tokenHash}`), and displays foreground notifications. Four Cloud Functions fire on Firestore `onCreate` events (notifications, broadcasts, announcements, meeting invites) to dispatch push payloads. Invalid tokens are cleaned up on delivery failure.

The v1 issues have been addressed: the duplicate `push` listener is gone (now uses `onBackgroundMessage` only), multi-device token storage uses a subcollection model, the chat notification blast function was removed entirely, and the logout race is fixed by capturing `userId` before nulling `currentUser`.

**Watch for:** Stale token cleanup is silently skipped for per-user notifications because `getTokensForUser` returns plain strings without document paths — expired tokens accumulate indefinitely for those paths (confirmed). The `cleanupFCMToken` on logout deletes *all* devices for the user rather than just the current device, so logging out on one device kills push for all other logged-in devices (confirmed). The VAPID key and `messagingSenderId` remain placeholders — a blocking deployment issue, though now clearly commented (confirmed).

**Verdict**: NEEDS_CHANGES

## High-level view

The token cleanup story has an asymmetry: `getAllTokens` (used by broadcast/pengumuman) returns objects with `docPath` populated, so stale token cleanup works. But `getTokensForUser` (used by per-user notifications and meeting invites) returns bare strings, and callers wrap them with `docPath: null`. When those tokens fail delivery, `sendToTokens` skips the cleanup. Stale tokens for individual users are never cleaned up through the functions path.

The logout cleanup deletes *all* device subcollection documents for the user, not just the current device's token. This contradicts the multi-device model: a user logged into phone and laptop who logs out on the laptop loses push on the phone until next login.

The VAPID key (`YOUR_VAPID_KEY_HERE`), `messagingSenderId` (`123456789`), and `gcm_sender_id` (`103953800507`) still don't form a consistent set. All three locations (core.js, service worker, manifest.json) must agree on the real project's sender ID before deployment.

## Token cleanup asymmetry in sendToTokens

The `sendToTokens` helper handles stale token removal only when the token entry carries a `docPath`:

```js
const tokenEntry = tokens[idx];
if (tokenEntry && tokenEntry.docPath) {
  batch.delete(db.doc(tokenEntry.docPath));
}
```

`getAllTokens` constructs the full path (`hrd_fcm_tokens/${userId}/devices/${deviceDoc.id}`), so broadcast and pengumuman notifications clean up expired tokens correctly. But `getTokensForUser` returns bare token strings, and both `onNotifikasiCreated` (for per-user targeting) and `onMeetingInviteCreated` wrap them as `{ token: t, userId, docPath: null }`. Failed deliveries to those tokens are logged but never cleaned from Firestore. Over time, users who cycle devices accumulate dead tokens that produce silent delivery failures on every targeted notification.

The fix is straightforward: have `getTokensForUser` return `{token, docPath}` tuples (it already has the `userId` parameter and `doc.id` available in the loop), or compute the path at the call site since the structure is deterministic.

## Logout removes all device tokens

```js
async function cleanupFCMToken(userId) {
  const devicesSnap = await db
    .collection("hrd_fcm_tokens").doc(userId).collection("devices").get();
  const batch = db.batch();
  devicesSnap.forEach((doc) => batch.delete(doc.ref));
  await batch.commit();
}
```

This deletes every device document for the user. The `hashToken` utility already exists in the client — the cleanup should compute `hashToken(currentToken)` before calling `messagingInstance.deleteToken()`, then delete only `hrd_fcm_tokens/{userId}/devices/{thatHash}`, preserving other devices. The `messagingInstance.deleteToken()` call already invalidates only the local browser's token, so the Firestore cleanup should mirror that scope.

## N+1 reads for role-based and broadcast targeting

`getAllTokens` fetches all user docs, then for each user queries their `devices` subcollection sequentially — O(users) Firestore reads per broadcast. `onNotifikasiCreated`'s role fallback does the same pattern filtered by `deviceData.role === targetUser`. At 100 users with 2 devices each, that's 101 reads per broadcast notification. Firestore collection group queries (`db.collectionGroup("devices").where("role", "==", targetRole)`) would reduce role-based targeting to a single indexed query. For broadcasts, `db.collectionGroup("devices").get()` would also work. This is a scalability concern — functional at small user counts but will hit latency and cost issues as the user base grows.
