# Firebase Cloud Messaging integration for IMS push notifications

This change adds push notification support to the IMS HR app via Firebase Cloud Messaging. The client side registers an FCM service worker, obtains a device token on login, stores it in Firestore, and displays foreground notifications. Five Cloud Functions fire on Firestore `onCreate` events (notifications, broadcasts, announcements, meeting invites, chat messages) to dispatch push payloads to registered tokens. Invalid tokens are cleaned up on delivery failure.

**Watch for:** The VAPID key is a placeholder string (`YOUR_VAPID_KEY_HERE`) — FCM token generation will silently fail in production until replaced (confirmed). The `gcm_sender_id` in `manifest.json` (`103953800507`) doesn't match the `messagingSenderId` in `firebaseConfig` (`123456789`), which will break push delivery on Android PWA installs (confirmed). The `onChatCreated` function sends a push to every registered user except the sender regardless of chat room/department scope, creating a notification blast problem at scale (confirmed).

## High-level view

The client-side integration in `core.js` follows a standard pattern: request permission, register the FCM service worker, get a token, store it, listen for foreground messages. The lifecycle is tied to login/logout but the logout cleanup is fire-and-forget (the async `cleanupFCMToken` call isn't awaited), so the local session is cleared before the Firestore delete completes — creating a race where `currentUser` is nulled before the function reads it.

The service worker (`firebase-messaging-sw.js`) handles background notification display and click-to-focus correctly, with one structural concern: it registers its own `push` event listener alongside the FCM SDK's `onBackgroundMessage` hook (initialized via `firebase.messaging()`), which means background notifications may be shown twice — once by the SDK's automatic handling and once by the explicit `push` listener.

The Cloud Functions are straightforward Firestore `onCreate` triggers with a shared `sendToTokens` helper that handles multicast delivery and stale token cleanup. The token storage model is one-token-per-user (the doc ID is the user ID), so a user who logs in on a second device overwrites their first device's token — only one device receives notifications at a time.

The configuration has two blocking issues: the VAPID key placeholder prevents token acquisition entirely, and the `gcm_sender_id` / `messagingSenderId` mismatch will prevent push delivery on PWA-installed Android instances.

7. **No Firestore security rules for hrd_fcm_tokens** — Any authenticated (or unauthenticated, depending on existing rules) client can read all FCM tokens. Add rules restricting token documents to their owning user.

## VAPID key and sender ID configuration

The VAPID key at line 20 of `core.js` is literally `"YOUR_VAPID_KEY_HERE"`. FCM's `getToken()` will throw (caught by the `try/catch`, logged as a warning, then silently ignored), so no client ever registers a token and no push notifications are delivered. This is a deploy-blocker.

Separately, `manifest.json` declares `"gcm_sender_id": "103953800507"` — a value the Chrome push infrastructure uses to route messages to the correct FCM project. But the Firebase config's `messagingSenderId` is `"123456789"`, a placeholder. If `103953800507` is the real sender ID, the `firebaseConfig` needs updating everywhere (core.js, firebase-messaging-sw.js). If it's also a placeholder, both need the real value. Either way, the mismatch will cause Android PWA pushes to fail.

## Duplicate push handling in the service worker

`firebase-messaging-sw.js` does two things that conflict:

```js
firebase.initializeApp(firebaseConfig);
const messaging = firebase.messaging();
```

Initializing `firebase.messaging()` in the service worker causes the FCM SDK to automatically handle background messages and display notifications. Then the file also adds:

```js
self.addEventListener("push", (event) => { ... showNotification(...) });
```

When a push arrives in the background, the SDK displays a notification via its internal `onBackgroundMessage` handler, and then the raw `push` event listener fires and calls `showNotification` a second time. The fix is to either use the manual `push` listener (and remove the `firebase.messaging()` call) or use the SDK's `messaging.onBackgroundMessage()` callback (and remove the manual listener).

## Single-token storage model

The token document uses `doc(currentUser.id).set(...)` — one document per user. If a user logs in on their phone and then on their laptop, the laptop's token overwrites the phone's. Only the most recent device gets pushes.

A more robust model stores tokens in a subcollection (`hrd_fcm_tokens/{userId}/devices/{tokenHash}`) or as an array field, so multiple devices are served. The Cloud Functions' `getTokensForUser` already returns an array, suggesting multi-device was anticipated, but the storage model doesn't support it.

## Chat notification scope

`onChatCreated` fetches all tokens except the sender's and sends a notification to every registered user in the system. If the chat is departmental (the collection is `hrd_chat` and the existing app mentions "Obrolan Divisi"), only members of the relevant department/room should be notified. At 50 active users sending 20 chat messages a day, this produces 1,000 unnecessary push notifications daily.

## Logout token cleanup race condition

```js
function doLogout() {
  cleanupFCMToken();  // async, not awaited
  currentUser = null; // nulled immediately
  ...
}
```

`cleanupFCMToken()` checks `if (!currentUser) return;` as its first line. If the JavaScript engine schedules the microtask continuation of `cleanupFCMToken` after `currentUser = null` executes (which is the default behavior — the awaits inside yield), the function returns early and the token remains in Firestore. After logout, that orphaned token continues receiving push notifications until it expires or the user logs in again and overwrites it.

## Firestore security for FCM tokens

The `hrd_fcm_tokens` collection stores device tokens with `role` and `userId` fields. There is no `firestore.rules` file in the repository. If Firestore is in test mode (allow all reads/writes), any client can enumerate all users' FCM tokens and roles. Even with basic auth rules, a logged-in user could read other users' token documents. The Cloud Functions use the Admin SDK (bypasses rules), but the client writes tokens directly — those writes need rules that enforce `request.auth.uid == resource.id` or equivalent app-level auth.
