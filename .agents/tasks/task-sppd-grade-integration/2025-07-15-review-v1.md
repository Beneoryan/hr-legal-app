# Grade-tiered benefit system for SPPD travel requests

This change introduces a grade-based benefit configuration layer to the existing SPPD (travel order) module. Four employee tiers (BOD, HEAD, SENIOR, STAFF) each get hardcoded allowance ceilings for daily stipend, hotel, transport, and meals. The SPPD form auto-fills cost estimates when dates change, warns users when entered amounts exceed their tier's limits, and persists the grade alongside each travel document. A bidirectional cross-reference links the SPPD record to its companion `hrd_dinas_luar` attendance record. On the portal side, employees see personalized benefit cards and a grade comparison table; admins see a read-only benefit matrix.

Watch for: Grade limits are advisory only — nothing prevents submission of above-limit amounts (confirmed). The routing layer calls the now-async `renderPortalPeraturan` and `renderPortalPerjalananDinas` without awaiting the returned promise, causing a visible render flash (confirmed). The `transportasiDiizinkan.join(', ')` and `transportasiDiizinkan[0]` insertions into innerHTML skip `escHtml`, which is safe today because the data is hardcoded, but becomes an XSS vector if these configs ever move to Firestore (likely).

## High-level view

The benefit config is a pair of hardcoded constants (`BENEFIT_CONFIG_BY_GRADE` for monetary limits, `PERATURAN_DINAS_BY_GRADE` for procedural rules) resolved at runtime via fuzzy string matching against `currentUser.gradeJabatan`. Both constants duplicate the same tier-matching logic in separate helper functions (`getGradeConfig` / `getGradePeraturan`), which is a maintenance risk if the grade vocabulary ever diverges between the two.

The SPPD form now calls `getUserGrade()` to fetch and cache the employee's grade from Firestore before rendering. Auto-calculated estimates populate cost fields based on trip duration, and `cekBatasGrade()` shows yellow warnings when amounts exceed tier ceilings — but the warnings are purely visual; `simpanSPPD()` saves the document regardless. This means grade limits function as guidance, not enforcement.

Cross-referencing between `hrd_perjalanan_dinas` and `hrd_dinas_luar` is now bidirectional: the SPPD document stores `dinasLuarId` and the dinas luar document stores `sppdId`. The link is created via a second Firestore write after both documents exist. If the second write fails, the SPPD document will lack its `dinasLuarId` field, leaving an orphan linkage.

The portal renders (`renderPortalPeraturan`, `renderPortalPerjalananDinas`) became async functions, but the route dispatcher in core.js calls them with `fn()` without `await`. The page shows stale or empty content until the grade query resolves and `innerHTML` is finally set.

## Grade resolution via fuzzy string matching

The two lookup helpers (`getGradeConfig`, `getGradePeraturan`) resolve a free-text `gradeJabatan` string into one of four tiers by checking whether the uppercase value contains keywords. The match order creates a bug for compound titles:

```js
if (g.includes('BOD') || g.includes('DIRECTOR')) return ...BOD;
if (g.includes('HEAD') || g.includes('MANAGER')) return ...HEAD;
if (g.includes('SENIOR') || g.includes('SUPERVISOR')) return ...SENIOR;
return ...STAFF;
```

"SENIOR MANAGER" hits the third branch, landing the employee in a lower tier than intended. The fix is to check compound patterns first, or normalize grades to enum values at write time rather than parsing at read time.

The identical matching logic is duplicated in both `getGradeConfig()` and `getGradePeraturan()`. If one is updated and the other isn't, an employee could see mismatched monetary limits vs. procedural rules.

## Grade limits are advisory, not enforced

`cekBatasGrade()` renders a yellow warning banner when manually entered amounts exceed the tier ceiling, but `simpanSPPD()` writes to Firestore unconditionally. The UX implies the limits are binding ("melebihi batas grade") when they are not.

For an internal HR tool where the document goes to an approver who can reject it, this may be acceptable — but at minimum a `confirm()` dialog asking "Estimasi melebihi batas grade, tetap ajukan?" would make the gap explicit to the user.

## Cross-reference lifecycle between SPPD and dinas luar

The new code establishes a bidirectional reference via three sequential writes:

1. SPPD document is written -> gets `sppdRef.id`
2. Dinas luar document is written with `sppdId: sppdRef.id` -> gets `dinasLuarRef.id`
3. SPPD document is updated with `dinasLuarId: dinasLuarRef.id`

Step 3 is a separate write. If it fails (network blip, permission error), the SPPD document lacks `dinasLuarId` while the dinas luar document correctly holds `sppdId`. The approval flow queries dinas luar by `noSPPD` string so it still works today — but any future code relying on `dinasLuarId` for navigation would break silently. A Firestore batch write would make this atomic at no extra cost.

## Unescaped template interpolation from hardcoded arrays

Several places insert `peraturan.transportasiDiizinkan.join(', ')` directly into innerHTML without `escHtml`, while adjacent values in the same template are escaped:

```js
<tr><td>Transportasi Diizinkan</td><td>${peraturan.transportasiDiizinkan.join(', ')}</td></tr>
<tr><td>Kelas Hotel</td><td>${escHtml(peraturan.kelasHotelDiizinkan)}</td></tr>
```

The inconsistency is a latent XSS risk if the admin "Konfigurasi Benefit" section ever evolves from read-only display to Firestore-backed editable config. Apply `escHtml` uniformly now to prevent the gap from becoming exploitable later.
