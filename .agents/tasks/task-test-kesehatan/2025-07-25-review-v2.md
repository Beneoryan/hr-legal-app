# Health Test (Test Kesehatan) module (v2)

A 627-line module adds a complete health test workflow to the HR app: admin schedules tests, fills a 7-section medical form, and reviews results; portal employees see only their own tests. Since v1, the portal query has been split into two targeted Firestore queries — one on `userId` and a scoped fallback on `userId == ''` for legacy records — eliminating both the full-collection download and the name-collision data leak. The schedule-save path now performs a name lookup to backfill `userId` when the admin types a name without using the dropdown.

Watch for: the name-lookup fallback in `simpanJadwalTestKesehatan` downloads the entire employee/candidate collection to find a match, reintroducing a smaller-scale full-scan on every schedule save (confirmed). The portal fallback query (`where('userId','==','')`) still matches records by name, so two employees with the same name and no `userId` will share health records — a narrower variant of the v1 issue that persists for legacy data (confirmed). The admin-side `showTestKesehatanTab` continues to fetch the entire `hrd_test_kesehatan` collection on every tab switch (confirmed, acceptable for admin-only context given low expected record count).

## High-level view

The portal privacy fix is well-structured: a primary `where('userId','==', u.id)` query covers the normal case, and a second query scoped to `userId == ''` handles legacy/candidate records with a client-side name check. The sensitive-data exposure is now limited to records that haven't yet been linked to a user ID, rather than the entire collection.

The `simpanJadwalTestKesehatan` name-lookup fetches the full `hrd_kandidat` or `hrd_karyawan` collection when the dropdown isn't used. This is an admin-only code path and the collections are small, but it's architecturally the same pattern the portal fix just eliminated. A `where('nama','==',nama)` query would be more consistent.

The admin table (`showTestKesehatanTab`) still does a full-collection `get()` on every tab switch and search keystroke. If the collection grows, every character typed in the search box triggers a full Firestore read.

## Portal query fix: from full-scan to targeted queries

The rewritten `loadPortalTestKesehatan` eliminates the v1 vulnerability:

```javascript
// Primary: exact userId match
const snapById = await db.collection('hrd_test_kesehatan').where('userId', '==', u.id).get();
snapById.forEach(d => { seen[d.id] = true; myTests.push({id: d.id, ...d.data()}); });

// Fallback: only records with empty userId, then name-check client-side
const snapByName = await db.collection('hrd_test_kesehatan').where('userId', '==', '').get();
snapByName.forEach(d => {
  if (seen[d.id]) return;
  const r = d.data();
  const matchNama = (r.nama || '').toLowerCase().trim() === (u.nama || '').toLowerCase().trim();
  if (matchNama) { seen[d.id] = true; myTests.push({id: d.id, ...r}); }
});
```

The fallback no longer exposes records belonging to other users (those have `userId` set to another person's ID). The remaining exposure is limited to records where `userId === ''` — newly scheduled tests for candidates whose names happen to match an employee. Once `simpanJadwalTestKesehatan` backfills the `userId` (via the name-lookup), these records exit the fallback pool. The window of exposure is between creation (if lookup fails) and the next time someone edits the record.

## Name-lookup full-scan in schedule save

When admin types a name without picking from the dropdown:

```javascript
if (!personId && nama) {
  const col = tipe === 'calon' ? 'hrd_kandidat' : 'hrd_karyawan';
  const lookupSnap = await db.collection(col).get();
  lookupSnap.forEach(d => {
    const rec = d.data();
    if (!personId && (rec.nama || '').toLowerCase().trim() === nama.toLowerCase().trim()) {
      personId = d.id;
    }
  });
}
```

This downloads the entire employee or candidate collection to find a name match. It also takes the first match when multiple people share a name, with no indication to the admin that multiple matches exist. A targeted query (`where('nama','==',nama)`) would reduce the read footprint and could surface ambiguity if multiple documents match.
