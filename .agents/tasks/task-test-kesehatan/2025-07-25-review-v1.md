# Health Test (Test Kesehatan) module for HR app

A new 602-line module adds a complete health test workflow: admin can schedule tests for candidates or existing employees, anyone with a scheduled test can fill out a comprehensive 7-section medical form, and results are stored in Firestore with status tracking and notification. The module integrates cleanly into the existing navigation and routing patterns.

Watch for: full-collection scan in the portal view matches records by name string comparison, which can return another person's health records if names collide (confirmed). The delete action is globally callable with no role check, so a console call deletes any record (confirmed, consistent with existing app pattern). The notification targets `'admin'` as a literal string rather than a user ID, which may be a dead letter depending on how notifications are queried (likely).

## High-level view

The module follows the established "global functions + Firestore + inline HTML" architecture. Admin gets a tabbed data table (Calon / Existing / Riwayat) with schedule buttons; portal employees see their own pending and completed tests. Integration is minimal: one script tag, two nav items, two route entries.

The portal view retrieves the user's tests by doing a full-collection `get()` then filtering client-side on name or userId. All documents transfer to the browser even if the user only owns one. Two employees with identical names see each other's medical records. The existing app uses the same pattern for payslips, but health data carries higher sensitivity.

An unrelated fix to `modules-peraturan.js` removes the "Daftar Isi" (table of contents) from the regulation renderer, leaving orphaned `<span id="bab-N">` anchors that nothing links to anymore.

## Portal record matching and data leakage

`loadPortalTestKesehatan` fetches the entire `hrd_test_kesehatan` collection and filters in JavaScript:

```javascript
const snap = await db.collection('hrd_test_kesehatan').get();
snap.forEach(d => {
  const matchNama = (r.nama || '').toLowerCase().trim() === (u.nama || '').toLowerCase().trim();
  const matchId = r.userId === u.id;
  if (matchNama || matchId) myTests.push({id: d.id, ...r});
});
```

Two problems compound here. First, every authenticated user downloads every health test document — Firestore client libraries transfer the full document set before client-side filtering. Second, the name match is case-insensitive and trimmed but not unique — "Ahmad Sari" scheduled as a candidate and "Ahmad Sari" who's an existing employee will share records.

The `userId` field is only populated for `tipe === 'existing'` (set from the dropdown's `value` in `simpanJadwalTestKesehatan`). For candidates, `kandidatId` is set instead, but the portal match checks `r.userId === u.id` — so candidates who later become employees won't see their earlier candidate test unless their name still matches.

## Notification routing

The save handler sends:

```javascript
sendNotification('admin', 'Test Kesehatan Disubmit', ...);
```

The `sendNotification` function stores `{ targetUser: 'admin', ... }` in `hrd_notifikasi`. The notification rendering typically queries `where('targetUser','==',currentUser.id)`. If that's the only query, notifications targeted at the literal string `'admin'` are only visible to a user whose Firestore document ID happens to be `'admin'`. The perjalanan-dinas module uses the same pattern (`sendNotification('admin', ...)`), so this is an app-wide ambiguity rather than a regression, but worth confirming it works as intended.

## Unrelated change: Peraturan table-of-contents removal

The diff strips the "Daftar Isi" block from `renderPeraturanHTML`. The old TOC's anchor targets (`<span id="bab-${bab.nomor}">`) remain inside each `<summary>` but are now dead — nothing in the UI links to them. Minor orphaned markup, no functional impact.
